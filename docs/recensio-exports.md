# Recensio export overview

(generated by AI)

This document summarizes the export-related code in:

- `src/recensio/plone/export.py`
- `src/recensio/plone/browser/export.py`
- `src/recensio/plone/browser/templates/export_rm.pt`
- `src/recensio/plone/browser/templates/export_rj.pt`

## High-level picture

The export code has three layers:

1. **Exporter classes** in `export.py`
   - collect review data
   - render XML or CSV
   - store generated files in the portal root
   - includes `ContainerExportView`, the minimal BrowserView used to render
     per-issue XML (lives here to avoid a circular import with `browser/export.py`)
2. **Browser views** in `browser/export.py`
   - trigger export-related actions
   - upload an already generated export
   - trigger DOI registration
   - reset export flags
3. **Registry settings** in the control panel
   - SFTP target for Chronicon uploads
   - filename prefix for uploaded files
   - DOI registration endpoint and credentials

There is also an important dependency on review-specific XML views elsewhere in
the package:

- `@@xml`
- `@@xml-lza`
- `@@xml-dara`

The exporters do not build metadata themselves. They call those views on review
objects and package the results.

## How the pieces hang together

### Chronicon flow

The Chronicon-related workflow is split into two steps:

1. **Generate** a portal-root file object named `export_metadata_xml.zip`
   using `ChroniconExporter`
2. **Upload** that existing ZIP to the remote Chronicon SFTP target using the
   `@@chronicon-export` browser view

So `@@chronicon-export` is an **upload view**, not the generator.

### Generic bulk export flow

`MetadataExport` in `browser/export.py` is a generic bulk-export runner. It:

- finds exporter factories for `IRecensioExporter`
- filters them with `needs_to_run()`
- iterates over published `Issue` and `Volume` objects
- collects published reviews below them
- feeds each review into each exporter via `add_review()`
- calls `export()` on each exporter at the end

In the **current package snapshot**, `MetadataExport` is exposed as a browser
view named `@@metadata-export` on the Plone site root. There are still no
visible ZCML registrations for the CSV or LZA exporters as `IRecensioExporter`
factories.

In contrast, the package **does** register a named utility factory for the
Chronicon exporter:

- `chronicon_exporter` -> `ChroniconExporterFactory`

So the current wiring is clearest for:

- the generic export runner `@@metadata-export`
- the Chronicon upload view `@@chronicon-export`
- the DOI update helper `@@dara_update`
- the LZA reset helper `@@reset-lza-export-flag`

## `export.py`

## `ContainerExportView`

A minimal `BrowserView` that wraps the per-issue XML template:

- template: `browser/templates/export_container_contextless.pt`
- called by `ChroniconExporter.finish_issue()` to render one XML file per issue
  or volume
- lives in `export.py` (not in `browser/export.py`) so that `browser/export.py`
  can safely import from `export.py` without creating a circular import

## `BaseExporter`

`BaseExporter` provides shared helpers:

- `get_export_obj(portal)`
  - looks up the generated export file object in the portal root
- `is_recent(export_xml_obj)`
  - treats an existing export file as recent if it was modified less than
    6 days ago

This is used by several exporters to decide whether a fresh export is needed.

## `ChroniconExporter`

Purpose:

- export review metadata as a ZIP file with **one XML file per issue or volume**

Important attributes:

- template: `browser/templates/export_container_contextless.pt`
- stored portal-root file name: `export_metadata_xml.zip`
- per-review XML source: `@@xml`

How it works:

- reviews are added in issue/volume order with `add_review(review)`
- each review is rendered through `review.restrictedTraverse("@@xml")()`
- when the current issue/volume changes, `finish_issue()` renders one XML file
  for that container and stores it in memory
- `export()` writes all gathered XML files into a ZIP archive and stores it as a
  `File` object in the portal root

Generated internal XML filenames look like:

- `recensio_<publication>_<volume>_<issue>.xml`
- or `recensio_<publication>_<volume>.xml` when there is no issue layer

Concurrency / freshness behavior:

- `needs_to_run()` returns false if a recent portal-root export already exists
- `running_export()` uses a temporary cache ZIP in the OS temp directory and
  treats an export as in progress for up to 1 hour

Return values from `export()` are status objects such as:

- file created
- current file already exists
- export already in progress

## `BVIDExporter`

Purpose:

- write a CSV mapping BVID values to review URLs

Stored portal-root file:

- `bvid_export.csv`

Behavior:

- includes only reviews with a BVID
- writes either the canonical URI or the normal absolute URL depending on the
  review configuration

## `MissingBVIDExporter`

Purpose:

- write a CSV listing reviews that still miss a BVID

Stored portal-root file:

- `missing_bvid.csv`

Behavior:

- stores `(UID, ISBN/ISSN)` pairs for reviews without a BVID

## `LZAExporter`

Purpose:

- export metadata **and full text PDFs** for long-term archive / deposit use

It subclasses `ChroniconExporter` and changes:

- stored portal-root file: `export_lza_xml.zip`
- per-review XML source: `@@xml-lza`

Additional behavior:

- includes PDFs inside the ZIP next to the XML metadata
- marks each exported review with the annotation key `LZA_EXPORTED`
- skips reviews that were already exported before

This makes LZA export **incremental** by default.

## `DaraExporter`

Purpose:

- collect `@@xml-dara` output for DOI registration-related export

Current state:

- the class exists, but `export()` is still a stub (`pass`)

So it looks like a scaffold or incomplete path rather than an active export
implementation.

## `register_doi_raw()` and `register_doi()`

These functions handle DOI registration / metadata update against the configured
da|ra endpoint.

They use registry settings for:

- `doi_registration_url`
- `doi_registration_username`
- `doi_registration_password`

Behavior:

- build XML from `@@xml-dara`
- send it via HTTP Basic Auth
- translate HTTP/IO/value errors into user-facing status/message pairs

Typical successful messages:

- `DOI successfully registered`
- `Metadata updated`

## `browser/export.py`

This module contains only **browser views**. It imports `LZAExporter` and
`register_doi` from `export.py`; the dependency is one-directional.

## `MetadataExport`

Purpose:

- generic bulk-runner for all registered `IRecensioExporter` exporters

What it does:

- uses a portal annotation key
  `recensio.policy.metadata_export_timestamp`
  as a coarse lock
- refuses to start if an export has run within the last 2 hours
- accepts a `force` request parameter to clear that lock first
- walks published issues/volumes and published reviews below them
- calls exporter `add_review()` for each review
- commits the transaction after each issue/volume to reduce memory pressure
- calls `export()` on each exporter and returns a HTML `<br />`-joined summary

Registration:

- ZCML name: `metadata-export`
- context: `Products.CMFPlone.interfaces.IPloneSiteRoot`
- permission: `zope2.View`

Usage URL:

```text
https://example.org/Plone/@@metadata-export
```

Notes:

- this is the browser entry point that orchestrates the bulk export run
- it supports a `force` request parameter to clear the in-progress timestamp
  guard before starting

## `@@chronicon-export`

Purpose:

- upload the current Chronicon ZIP from the portal root to a remote SFTP server

Registration:

- ZCML name: `chronicon-export`
- context: `Products.CMFPlone.interfaces.IPloneSiteRoot`
- permission: `cmf.ManagePortal`

Usage URL:

```text
https://example.org/Plone/@@chronicon-export
```

Configuration used:

- `xml_export_server`
- `xml_export_username`
- `xml_export_password`
- `xml_export_filename_prefix`

Behavior:

1. read SFTP settings from the registry
2. look up `export_metadata_xml.zip` in the portal root
3. open an SFTP connection to the configured host on **port 2222**
4. upload the ZIP
5. name the remote file as:

```text
[prefix]_[DDMMYY]_all.zip
```

Example:

```text
recensio_050426_all.zip
```

Important limitation:

- this view only uploads an already prepared ZIP
- it does **not** regenerate `export_metadata_xml.zip`

Typical return values:

- `no host configured`
- `Could not get export file object: export_metadata_xml.zip`
- `Export successful`
- `Export failed, IOError: ...`
- `Export failed, SSHException: ...`
- `Export failed, <sent>/<total> bytes transferred`

## `DaraUpdate`

Purpose:

- register or update DOI metadata for the current object through da|ra

Behavior:

- only performs work on `POST`
- calls `register_doi(self.context)`
- stores the result as a Plone status message
- redirects back to the current object afterwards

Registration:

- ZCML name: `dara_update`
- context: `recensio.plone.interfaces.IReview`
- permission: `cmf.ModifyPortalContent`

Usage pattern:

- intended for review objects
- only does work on `POST`
- redirects back to the review after execution

## `ResetLZAExportFlag`

Purpose:

- recursively clear the `LZA_EXPORTED` annotation so objects can be exported
  again by `LZAExporter`

Behavior:

- walks the current context recursively
- for any object providing `IReview`, resets the LZA export flag to `False`
- returns a plain text list of affected object paths

Registration:

- ZCML name: `reset-lza-export-flag`
- context: `*`
- permission: `cmf.ModifyPortalContent`

Usage pattern:

- can be called on a subtree to clear LZA export flags recursively
- returns a plain text list of affected paths

## Practical summary

If you want to run the export generation and Chronicon upload path, the
practical flow is:

1. call `@@metadata-export` on the site root to generate or refresh export
   artifacts
2. ensure the Chronicon SFTP settings are configured
3. call `@@chronicon-export` on the site root as a manager

If you are working on the broader export subsystem, keep in mind:

- `ChroniconExporter`, `LZAExporter`, BVID exporters, and DOI helpers live in
  `export.py`
- `ContainerExportView` also lives in `export.py` to avoid a circular import
- `MetadataExport` is the bulk orchestration layer and is now exposed as
  `@@metadata-export`
- `ChroniconExport` handles only the remote SFTP upload step
- `DaraUpdate` and `ResetLZAExportFlag` are also exposed as browser views
- some exporter classes still exist in code without obvious factory
  registrations in the current package ZCML

## Export templates

### `export_rm.pt` — review of monograph

Used for reviews of books/monographs. Fields are read directly as Dexterity
attributes (Python 3 / Plone 6 — no AT getter methods):

| XML element | Context attribute | Behavior |
|---|---|---|
| `isbn` | `context/isbn` | `IBookReview` |
| `language_reviewed_object` | `context.languageReviewedText` | `ITextReview` |
| `pages` | `context/pages` | `ISerial` |
| `place_of_publication` | `context/placeOfPublication` | `IPrintedReview` |
| `publishing_company` | `context/publisher` | `IPrintedReview` |
| `series` | `context/series` | `ISerial` |
| `subtitle` | `context/subtitle` | `IBookReview` |
| `volume_number` | `context/seriesVol` | `ISerial` |
| `year` | `context/yearOfPublication` | `IPrintedReview` |
| `bvid` | `context/bv` | `IBase` |
| `classification_place` | `context.ddcPlace` | `IBase` |
| `classification_subject` | `context.ddcSubject` | `IBase` |
| `classification_time` | `context.ddcTime` | `IBase` |
| `doi` | `context/doi` (condition: `doiRegistrationActive`) | `IBaseReview` / `ISettingsDoi` |
| `language_review` | `context.languageReview` | `IBase` |

| `page_last` | `context/pageEndOfReviewInJournal` | `IPagesOfReviewInJournal` |

### `export_rj.pt` — review of journal

Used for reviews of journals / journal articles. Follows the same Dexterity
attribute pattern as `export_rm.pt`.  Additional journal-specific fields:

| XML element | Context attribute | Behavior / field |
|---|---|---|
| `reviewed_journal_editor` | `context/editor` | content type field |
| `reviewed_journal_issn` | `context/issn` | `IJournalReview` |
| `reviewed_journal_issue` | `context/issueNumber` | `IJournalReview` |
| `reviewed_journal_publisher` | `context/publisher` | `IPrintedReview` |
| `reviewed_journal_pubplace` | `context/placeOfPublication` | `IPrintedReview` |
| `reviewed_journal_pubyear_real` | `context/yearOfPublication` | `IPrintedReview` |
| `reviewed_journal_pubyear_fictive` | `context/officialYearOfPublication` | `IJournalReview` |
| `reviewed_journal_title_short` | `context/shortnameJournal` | `IJournalReview` |
| `reviewed_journal_volume` | `context/volumeNumber` | `IJournalReview` |

The `reviewer` block, `bvid`, `doi`, classification fields, `page_first` /
`page_last`, `filename`, `fulltext`, and `originalurl` are identical to
`export_rm.pt`.

### Note on `reviewAuthors`

`reviewAuthors` (defined on `IBase`) is a Dexterity `RelationList` of
`RelationValue` objects pointing to `Person` content items. Templates must go
through `to_object` to reach the actual person:

```xml
<reviewer tal:repeat="author context/reviewAuthors">
    <reviewer_last_name tal:content="author/to_object/lastname"/>
    <reviewer_first_name tal:content="author/to_object/firstname"/>
</reviewer>
```
